# Haystack EVE-Style UI — Functional Requirements & Verification

> Status: contract / source of truth for the `src/client/App.tsx` rewrite.
> Stack: Bun · strict TypeScript · Hono · SQLite · React Three Fiber + Three.js · WebSocket replication · stateful CLI client · Playwright e2e.
> Verification harness: the existing `tests/e2e/*.ts` (raw `Bun.spawn` server+client, CLI-driven state, `chromium` from `playwright`, plain `throw`-based assertions). This document does **not** assume `@playwright/test`.

## 1. Purpose & Scope

This document is the complete, self-contained functional-requirements spec for Haystack's EVE-Online-inspired UI. It merges the per-area research with the adversarial audit: every audit fix is applied, the completeness critic's missing requirements are folded in, every item the decision-compliance auditor flagged as out-of-scope is deleted, and overlapping requirements across areas are reconciled into one numbered set.

### 1.1 The three binding decisions (absolute scope rules)

**DECISION 1 — Newtonian + EVE chrome ONLY.** Flight stays exactly as the goal states: manual zero-g Newtonian impulse thrust (directional/axis buttons emitting `{type:'input',pilotId,clientTick,command:{impulse}}`) plus a single Stabilize action (`command:{impulse:{0,0,0},stabilize:true}`) that costs heat. EVE's **navigation & targeting model is OUT**: no click-to-move, no Approach/Orbit/Keep-at-Range/Align/Warp/auto-pilot, no target-lock-to-activate-modules combat loop, no module-cycle/capacitor-drain activation. We adopt ONLY EVE's **window + chrome UX** and its **selection/inspection affordances** (select-to-inspect, an Overview-style sortable list, a Selected-Item panel, right-click context menus exposing Haystack's existing verbs). Those menus must never introduce orbit/warp/lock/auto-pilot.

> **EVE-accuracy correction (applied throughout):** In real EVE, **single-click** selects a row/bracket; **double-click** on a row, an in-space bracket, or empty space is a **navigation** command (align/approach). Haystack therefore binds selection to **single-click** (EVE-accurate inspection) and **intentionally does not** reproduce EVE's double-click-to-align. Every double-clickable surface is guarded by a tested negative invariant (X-04): a double-click emits **zero** `input` WS frames.

**DECISION 2 — Core window subset, existing features only.** The shippable windows map 1:1 to real Haystack features: Flight/Ship-HUD (manual thrust), Scanner (= the Overview), Cargo/Inventory, Comms/Chat (+ Local pilot list), Character cards / Show-Info, Bases/Structures (+ station/upgrade services, with Upgrades as a **panel inside Bases**, not a separate window). EVE windows with no Haystack feature behind them are **forbidden**: no Fitting, Market/Industry, Drones, Fleet hierarchy, PI. The **Map is P2 "when practical"**, not core.

**DECISION 3 — Functional + verifiable visual.** Every visual/aesthetic requirement is written so it can be asserted in an automated e2e/screenshot test: computed-style luminance/opacity thresholds, HSL hue bands, bounding-box geometry, DOM attribute flags, pilot-scoped `localStorage` keys, captured WS-frame shapes, pixel sampling of the canvas. No untestable "make it pretty."

### 1.2 EVE visual reference of record

The visual reference is EVE's current **Photon UI** (the 2022+ overhaul), **not** the dated "Carbon" engine/UI-framework name. Photon's relevant, grounded facts: a default dark theme; per-window transparency controls (gated on "Blur the background"); and a separate light-background toggle which Haystack does **not** expose. Haystack pins **one fixed palette** rather than reproducing Photon's user tint customizer (Decision 3 verifiability).

### 1.4 Priority scheme

**P0** = blocks ship; **P1** = required but not blocking the P0 floor; **P2** = "when practical" nice-to-have. The P0 floor is: first-paint-is-the-game, dark-translucent chrome that never fully occludes the scene, movable + resizable + minimize + close/reopen-via-Neocom windows, layout persistence, the Overview (sort/filter/single-click-select), Selected-Item sync, right-click menus with banned-verb absence, chat round-trip, the mine→sell→upgrade loop, the manual-thrust HUD, the data-testid convention, and usability at 1920×1080 + sane mobile degradation. Pin/opacity/lock/snap/stack/blink (WM-08, WM-10, WM-11, WM-12) and the Map are explicitly subordinate enhancements that must **not** block the P0 floor or appear in the Definition of Done.

## 9. E2E Verification & Testing Plan

### 9.1 Harness paradigm (decide once, applies to all snippets)

The existing harness (`tests/e2e/screenshot.ts`, `tests/e2e/multiplayer.ts`) is **raw `Bun.spawn` scripts** importing `chromium` from `playwright`, asserting with plain `throw new Error(...)`. There is **no `@playwright/test`** runner — no `expect`, `expect.poll`, `toHaveCount`, `toBeVisible`. The new specs **adopt the same raw style** for consistency. A small shared helper module `tests/e2e/helpers.ts` provides the test-runner ergonomics as plain functions:

```ts
// tests/e2e/helpers.ts
export function assert(cond: unknown, msg: string): asserts cond {
  if (!cond) throw new Error(`Assertion failed: ${msg}`);
}
export async function pollUntil<T>(
  fn: () => Promise<T>,
  ok: (v: T) => boolean,
  timeoutMs = 8000,
  stepMs = 150,
): Promise<T> {
  const deadline = Date.now() + timeoutMs;
  for (;;) {
    const v = await fn();
    if (ok(v)) return v;
    if (Date.now() > deadline) throw new Error("pollUntil timed out");
    await new Promise((r) => setTimeout(r, stepMs));
  }
}
export async function count(page: import("playwright").Page, sel: string): Promise<number> {
  return page.locator(sel).count();
}
// Pre-navigation WS + request capture (X-15). MUST be called before page.goto.
export function captureTraffic(page: import("playwright").Page) {
  const wsSent: string[] = [];
  const requests: { url: string; body: string | null }[] = [];
  page.on("websocket", (ws) => ws.on("framesent", (f) => wsSent.push(String(f.payload))));
  page.on("request", (r) => {
    if (/\/api\/ships\/[^/]+\/(scan|mine|thrust)/.test(r.url()))
      requests.push({ url: r.url(), body: r.postData() });
  });
  return {
    inputFrames: () =>
      wsSent
        .map((p) => {
          try {
            return JSON.parse(p);
          } catch {
            return {};
          }
        })
        .filter((m) => m.type === "input"),
    requests,
  };
}
```

Every snippet below uses these helpers; they are copy-paste runnable on `bun tests/e2e/ui.ts`.

### 9.4 Visual / screenshot baseline approach

- The WebGL scene is **non-deterministic** (seeded starfield, GPU variance). The existing harness deliberately only asserts `uniqueColors >= 2`. **No full-frame pixel baseline over the canvas.**
- Visual regression = the **deterministic computed-style palette/luminance audit** (VIS-13) emitted as JSON in the screenshot pipeline, plus optionally a **canvas-masked** chrome-only diff (or a `?hideScene=1` solid-sentinel render with fixed seed and documented per-pixel tolerance).
- Committed reference (if used) lives at `screenshots/baseline/haystack-chrome.png` (canvas masked), distinct from the per-run output `screenshots/haystack-e2e.png`.

## 10. Definition of Done

Each P0 capability is "done" only when its named check is **green** in `verify:e2e`. The two negative guards are mandatory so Decision 1 and Decision 3 cannot be quietly violated.

- [ ] **Canonical roster (X-01):** exactly six `window-*` roots; exactly six `neocom-*` buttons; no Fitting/Market/Industry/Drones/Fleet/PI/Upgrades window or Neocom entry. _Human: opening the Neocom shows six launchers and nothing phantom._
- [ ] **First paint = game (X-06):** direct to `haystack-app` + non-blank canvas, no click-through, no marketing.
- [ ] **Dark translucent chrome (VIS-01/02):** all window bodies luminance < 0.18, alpha < 0.95; no UI surface > 0.85 luminance. _Human: windows are tinted glass over space, never opaque cards._
- [ ] **Overlay budget (X-07/X-14):** default-open windows cover < 65% of 1920×1080; scene visible.
- [ ] **Movable + resizable (WM-01/WM-02):** title-bar drag moves; 8-handle resize grows/shrinks with min clamp.
- [ ] **Minimize / close / reopen (WM-03/WM-04/WM-05):** minimize collapses + restores; close detaches; Neocom reopens.
- [ ] **Layout persistence (WM-09) + re-clamp (X-13):** geometry survives reload under `haystack.layout.<pilotId>`; reset restores defaults; restored layout re-clamps to a smaller viewport.
- [ ] **Overview (OVR-01/02/03):** real table; default distance-asc sort; type filters hide/show.
- [ ] **Single-click select, inspect-only (OVR-04):** one selected row; **zero `input` frames** on click/dblclick; position/velocity unchanged.
- [ ] **Selected-Item sync + actions (OVR-06/07):** panel mirrors selection; action row maps only to existing verbs; mine raises cargo; scan sends `targetAsteroidId`.
- [ ] **Right-click menus + banned-verb absence (OVR-11/12, X-03):** allowed verbs present; **no** orbit/approach/warp/align/keep-at-range/lock/jump/module item (testid or text).
- [ ] **No double-click navigation / no aim-thrust (X-04):** double-click on row/bracket/empty space emits zero `input` frames; no heading/aim-assisted thrust affordance.
- [ ] **Manual-thrust HUD (HUD-01/02/03, X-02/X-05):** module-style thrust buttons emit `{impulse}`; Stabilize emits `{impulse:0,stabilize:true}` and strictly increases heat; heat gauge never auto-drains; only `subscribe`/`input` WS message types.
- [ ] **Mine → sell → upgrade loop (CARGO-01/02/03, BASE-04):** cargo bar + itemized hold update; sell raises credits + empties; upgrade raises the stat + drops credits.
- [ ] **Comms (COMMS-01/02/03):** channel-filtered log; Local list + head count; Enter-to-send; DM carries `toPilotId`.
- [ ] **Reticle is cosmetic (VIS-04):** centered, `pointer-events:none`, not wired to a thrust vector.
- [ ] **data-testid convention (X-08) + helper (X-15):** documented, unique, `haystack-app` + `.pilot-card` preserved; pre-goto WS/request capture helper exists.
- [ ] **Desktop + mobile (X-09):** all default windows in-viewport at 1920×1080; no overflow + non-blank canvas at 390×844.
- [ ] **Backward-compat (X-17):** `verify:multiplayer` and `verify:screenshot` still pass standalone on the new build.
- [ ] **CI wiring (E2E):** `verify:ui` added; `verify:e2e` runs all three and exits non-zero on any failure; no orphaned processes after teardown.

> P1/P2 items (Show-Info, marker, brackets, context-menu polish, Comms scrollback/unread, Character roster, Bases list/deploy/field-index, keyboard, tooltips, loading/busy states, pin/opacity/lock/snap/stack/blink, Map) are tracked in the Requirement Index but are **not** gates on the P0 floor.
