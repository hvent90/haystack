## Codebase Patterns
- The asteroid field is client-derived (src/client/eve/field-derivation.ts) and a pure function of FieldSummary {seed, cellSize, totalAsteroids, renderedLimit}; the server (src/server/field.ts) only streams ~30 seeded rocks + FieldSummary. Server publishAll cost is independent of renderLimit (measured ~0.18ms p95 at 2k and 100k). DO NOT reintroduce per-tick server field work.
- Client render cap lives in src/client/eve/components/WorldView.tsx: ASTEROID_CAPACITY = 2000, and the InstancedAsteroids component uses Math.min(asteroids.length, ASTEROID_CAPACITY). One InstancedMesh + one bounding sphere => Three.js culls all-or-nothing (no per-instance frustum culling).
- renderedLimit flows server->client via FieldSummary.renderedLimit, set from HAYSTACK_RENDERED_LIMIT env on the server (src/server/field.ts:renderedLimit()).
- e2e harness uses Playwright/chromium; tests/e2e/screenshot.ts shows the boot pattern: Bun.spawn(["bun","src/server/main.ts"]) + Bun.spawn(["bunx","vite",...]) then chromium.launch(); waits for [data-testid='haystack-app'] then a canvas. Reuse this for the client-render benchmark.
- Field generators MUST stay in exact parity: scripts/bench/parity-check.ts compares server streamedFieldAsteroids vs client deriveVirtualField. Touch ids/positions/order => fix both sides in one commit.
- Real-hardware baseline (user report): 100k asteroids rendered at < 1 FPS on a 2025 MacBook Pro. The render path (full 100k instance submission, no culling, full rebuild per cell-cross) is the bottleneck — not the server.
- Client render stats are app-exposed via window.__HAYSTACK_RENDER_STATS__ (src/client/eve/render-stats.ts, published each frame from RenderDriver in WorldView.tsx). submittedInstanceCount is accumulated from the asteroid InstancedMesh's onAfterRender, GATED ON MATERIAL IDENTITY (renderedMaterial === the asteroid MeshStandardMaterial). three only calls onAfterRender for objects that survive frustum culling; the asteroid material is used ONLY in the main color pass (shadow pass -> depth material, post NormalPass -> normal material), so the gate counts each in-view instance exactly once/frame and will naturally SUM across per-cell sub-meshes once the field is chunked (C3). This is the robust culling signal — do not replace with renderer.info.
- renderer.info.render auto-resets on every internal render() call and the postprocessing EffectComposer issues MANY per frame, so a naive read returns only the last fullscreen quad. RenderDriver sets gl.info.autoReset=false and reads+resets once per frame to get whole-frame totals (drawCalls/renderedTriangles across shadow+normal+color+post; ~3x the asteroid color-pass tris). The asteroid-specific asteroidDrawCalls/asteroidTriangles (submitted * 36 tris for dodecahedron detail-0) are the precise per-field numbers.
- window.__HAYSTACK_RENDER_DEBUG__ = { reset(), faceAway(on) }. faceAway rotates the camera 180° (RenderDriver applies a yaw180 quaternion) so the benchmark can measure an empty-view submitted count. No effect in normal play (nothing sets it). NOTE: from the field CENTER the camera sits INSIDE the field's bounding sphere so facing-away still submits everything at baseline; making cameraFacingEmpty ~0 (C3) will require positioning the ship at the field EDGE looking outward.
- client-render benchmark: scripts/bench/client-render.ts. Per HAYSTACK_RENDERED_LIMIT it boots server+vite+chromium, joins a pilot (CLI), loads the client, waits for the derive, then samples 3 phases: steady (facing field), faceAway (empty), and a thrust-burst movement phase for worstCellCrossFrameMs. Plus an in-process deriveVirtualField micro-bench at 100k. Knobs: BENCH_LIMIT_GRID, BENCH_PORT, BENCH_CLIENT_PORT, BENCH_HEADFUL. Thrust is driven via POST /api/ships/:id/thrust (impulse clamped to mag 12 per call, so it sends a burst to build speed).

Fails identically with our client changes stashed. Server-side field-diagnostic test, unrelated to the client render path. 54 pass / 1 fail.
